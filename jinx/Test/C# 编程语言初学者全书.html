.NET 简介

正如我们在导言部分简要讨论的那样，.NET 提供了一个运行 C# 应用程序的环境。

.NET 的一个基本功能是可以描述为称为 CoreCLR 或 Core Common Language Runtime 的虚拟机。

Core Common Language Runtime 提供 Just-in-time 编译、内存管理、垃圾回收、安全性和异常处理等服务。.NET 还提供了各种基类库，这些库提供可由 C# 代码利用的通用功能。

.NET 的第一个版本是 2002 年发布的 .NET Framework。.NET Framework 只能在某些 Windows 平台上运行，并且必须完全安装在目标计算机上。

.NET Core 于 2016 年发布，提供了针对云进行了优化的模块化跨平台版本的 .NET。.NET Core 的一个重要功能是，只需将应用程序使用的依赖项发送到目标计算机，这与 .NET Framework 必须完整存在于目标计算机上不同。

.NET 的这两个版本（.NET Framework 和 .NET Core）的快速发展导致 .NET 的碎片化不断增长。

为了处理 .NET 的持续碎片化，Microsoft 创建了 .NET Standard，其中运行 .NET 的所有平台都必须支持 .NET Standard。这是统一 .NET 的第一步，但只是一个临时解决方案。

然后在 2020 年 11 月，.NET 5 发布。此版本的 .NET 保留了 .NET Framework 和 .NET Core 的强大功能，但此版本的重要意义在于 .NET 5 意味着 .NET 现在统一在一个保护伞下（就像它一样）。现在不再有 .NET Framework 和 .NET core，而只是一个 .NET 版本向前发展。

随着次年（即 2021 年 11 月）.NET 6 的发布，出现了许多重大改进和新功能。也许 .NET 6 最重要的一点是，它巩固了 .NET 的统一。

目前，.NET 是一个跨平台、模块化、敏捷、快速、可靠且安全的环境，您的 C# 应用程序可以在其中运行。这意味着 C# 和 .NET 现在已经发展到可以“一次编写，随处运行”的地步。

值类型和引用类型

C# 数据类型可以分为两个主要分类：值类型和引用类型。这些主要数据类型分类表示 C# 数据类型的数据在内存中的存储方式。

值类型存储在称为堆栈的内存位置，其中分配给变量的值存储在堆栈上的相关内存空间中。

引用类型存储在称为堆的内存位置，其中实际数据存储位置的地址驻留在堆栈上，并指向实际数据存储在堆上的位置。

存储在堆栈上的数据和存储在堆上的数据之间的一个关键区别是，存储在堆栈上的所有数据都具有固定大小，而存储在堆上的数据没有固定大小。与管理堆上存储的数据相比，堆栈上存储的离散数据的固定大小意味着存储和检索此类数据的效率更高。

C# 中的对象数据类型是 C# 中所有数据类型的根类型。对象数据类型是引用类型，因此直接从对象数据类型继承的所有类型都是引用类型。

简单来说，在C#中，当你将一个引用类型（如类实例）赋值给另一个变量时，实际上是复制了对同一块堆内存区域的引用，而不是复制这块内存区域的内容。因此，任何通过其中一个引用来做的修改都会影响到另一引用所指向的对象状态。

C# 字符串的不变性

字符串是引用类型，这意味着对内存地址的数字引用存储在堆栈上，并指向存储在堆上的实际字符串数据。

字符串引用类型与其他引用类型（例如，从类实例化的对象）之间的区别在于，特定字符串（存储在堆上）的数据不能直接在内存中更改。这意味着，例如，每次在代码中发生串联作时，存储在堆栈上的内存地址都会被简单地修改为指向堆上的新内存位置，该位置存储由于相关串联作而创建的新字符串。

常量和只读变量

常量简介

常量就像一个变量，你可以通过声明它并为其分配一个值来在其中存储一个值。然后，您可以使用用户可读的名称来引用该值，该名称表示代码中的 const 值。

常量与变量的不同之处在于，必须在声明它的同一行上为其分配一个值。此外，一旦为常量分配了值，就不能在代码中的任何其他位置为该常量分配新值。

您应该在代码中使用常量，使代码更具可读性和可维护性。适当使用 const 时，不必重复在代码中分配给 const 的值。当您需要在代码中引用该值时，您可以改为在代码中包含您为 const 提供的人类可读名称，以表示相关的常量值。

如果需要更改 const 值，您只需在一个位置更改代码（即声明 const 的位置）。此更改将自动传播到其他代码行（范围适当）中引用常量的位置。

只读变量简介

只读变量与变量类似，您可以通过声明值并为其分配值来在其中存储值。然后，您可以使用表示只读变量的人类可读名称来引用该值。

只读变量与变量的不同之处在于，其值在声明后只能在代码中更改一次。它可以在类的构造函数中更改其值，但不能在代码的其他部分中更改。

因此，如果您为只读变量分配了在声明该变量的行中具有值，则可以在类的构造函数中为该只读变量分配不同的值，但不能在任何其他代码中为该只读变量分配新值。

因此，只读变量通常称为运行时常量。在编译时，在同一行代码上声明一个常量并为其赋值，并且该 const 的值随后无法在编译时更改（因此也不能在运行时更改）。

为只读变量分配的值随后无法在运行时更改。因此，对于只读变量，在类的构造函数中为其分配一个值，在运行时从该类创建对象实例时，将设置一次。进行此赋值后，无法更改该只读变量。

C 语言中的反射

反射是 C# 中的一个概念，程序员能够创建可以在运行时动态读取 .NET 程序集元数据的 C# 代码。这为您提供了一个强大的工具，您可以在其中创建代码，以便在运行时根据程序集的元数据动态分析 .NET 程序集。例如，您的代码可以使用 Reflection 读取程序集的相关元数据，并输出有关所分析程序集的描述性元数据。

反射甚至可用于在运行时将对象动态绑定到驻留在程序集中的特定类型，并执行驻留在该类型方法中的代码。这称为后期绑定。

大多数情况下，您不会使用反射来调用 .NET 程序集中的代码，而是使用早期绑定。早期绑定意味着 C# 编译器在编译时知道程序集的相关程序元素，例如程序集的类和公共方法。

使用早期绑定是使用类型功能的最安全方式，因为在编译时会标记与早期绑定对象中的调用代码相关的任何潜在异常。这可以防止将可能错误的代码发布到生产环境中，从而导致发生运行时错误。

使用早期绑定时，由于 .NET 程序集的自描述性质，其中元数据存储在 .NET 程序集中，因此 C# 编译器能够在编译时知道有关程序集的所有相关详细信息。

因此，存储在 .NET 程序集中的此元数据意味着可以进行早期绑定，其中 C# 编译器在编译时具有所有必要的类型知识（如果需要）。

使用反射，您可以使用在运行时发生的后期绑定技术。后期绑定可用于在运行时动态绑定到对象（派生自驻留在相关程序集中的类型），并使用驻留在相关 .NET 程序集中的功能。

这是一个功能强大的工具，但不是在 .NET 程序集中执行功能的安全方法，因为后期绑定意味着调用代码在运行时通过读取目标程序集的元数据来了解目标程序集，然后再在程序集中执行代码。因此，后期绑定技术更容易出现运行时错误，因为无法在编译时（即在运行时执行程序集中的代码之前）处理潜在错误。
